## 2025-12-19: Dice Fraction Delta Reward

**Change**: Added `dice_fraction_scale` reward parameter that provides non-linear reward at round end based on change in dice fraction.

**Rationale**: Provides intermediate reward signal between sparse game-end and dense step-level rewards. The delta-based approach rewards transitions (not states), and naturally weights late-game rounds more heavily since each die represents a larger fraction.

**Implementation**:
- At round end, computes `delta = (own_dice/total_after) - (own_dice/total_before)`
- Reward = `dice_fraction_scale * delta`
- Works for all players including the round loser and eliminated players
- CLI args: `--dice-fraction-scale`, `--game-win`, `--elimination`

**Key Properties**:
- Zero-sum: Winner gains fraction, loser(s) lose fraction
- Non-linear by phase: 1 die out of 3 total vs 1 die out of 9 total
- State reconstructed from round result (loser lost 1 die)

**Results** (3 players, 1 DQN vs 2 Heuristic, 3 dice, 1000 episodes):

| Configuration | DQN Win Rate | Expected |
|---------------|--------------|----------|
| dice_fraction_scale=1.0 only | **21.6%** | 33.3% |
| sparse + dice_fraction_scale=1.0 | 12.6% | 33.3% |
| sparse only (baseline) | 7.0% | 33.3% |

**Observations**:
- Dense intermediate feedback accelerates early learning significantly
- dice_fraction_scale alone outperformed sparse in short training
- Combined rewards may need scale tuning to avoid interference
- Longer training runs needed to assess asymptotic behavior

## 2025-12-19: Composable Reward Structure

**Change**: Implemented 3-layer composable reward system via `RewardConfig` dataclass.

**Rationale**: Sparse game-end rewards have high variance. Denser rewards can accelerate learning but risk reward hacking. Composable structure allows experimentation.

**Layers**:
1. **Game-level** (sparse): `game_win=1.0`, `elimination=-1.0`
2. **Round-level** (semi-sparse): `round_win`, `round_loss`, `survive_round`
3. **Step-level** (dense): `successful_call`, `failed_call`, `got_called_truthful`, `got_called_bluffing`

**Presets**:
- `sparse`: Game-end only (default, backward compatible)
- `round`: + round rewards (0.1/-0.1)
- `dense`: + step rewards (0.05/-0.05 for calls)

**CLI args**:
- `--reward-preset {sparse,round,dense}`
- `--reward-round-win FLOAT` (override)
- `--reward-successful-call FLOAT` (override)

**Results** (3 players, 1 DQN vs 2 Heuristic, 3 dice, 5000 episodes):

| Preset | Final Win Rate | Expected |
|--------|---------------|----------|
| Sparse | **35.2%** | 33.3% |
| Dense  | 32.4% | 33.3% |

Sparse outperformed dense. Possible explanations:
- Dense rewards add noise, complicating credit assignment
- Step rewards may encourage sub-optimal behaviors (calling too often?)
- Reward magnitudes need tuning (0.05 for calls may be too high)

Further experiments needed with different reward scales

## 2025-12-19: Dual Replay Buffer for Off-Policy Learning

**Change**: Added separate replay buffer for opponent (heuristic) transitions.

**Rationale**: DQN is an off-policy algorithm, so it can learn from transitions generated by any policy. Storing opponent transitions provides:
- More data per episode (roughly doubles transitions collected)
- Diverse exploration of state space
- Learning from a competent player's behavior

**Implementation**:
- `collect_episode` now tracks state/action for all agents, not just DQN
- Transitions routed to `dqn_buffer` or `opponent_buffer` based on seat
- `train_step` samples from both buffers with configurable ratio
- New CLI arg: `--opponent-ratio` (default 0.2)

**Hyperparameters**:
- `opponent_ratio`: Fraction of batch from opponent buffer (0.0-1.0)
- Can be scheduled like epsilon if desired

**Results** (3 players, 1 DQN vs 2 Heuristic, 3 dice, 5000 episodes):
- Final eval: 22.8% win rate (expected 33.3% if equal skill)
- Steady improvement: 0% -> 24% over training
- Still below expected, but similar to previous runs without opponent buffer
- Need more episodes or tuning to determine if opponent buffer helps

## 2025-12-18: Multi-DQN Seat Support

**Change**: Allow multiple seats to be controlled by the same DQN policy during training.

**Rationale**: Self-play style training where multiple seats share the same policy can improve learning by:
- Seeing the game from different positions
- More DQN transitions per episode

**Implementation**:
- `dqn_seats` parameter specifies which seats use DQN
- Seats randomized each episode for position-agnostic learning
- New CLI arg: `--num-dqn`
